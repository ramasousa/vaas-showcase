01

Visão Geral

Este documento converte o brainstorm e os protótipos já validados em plano de execução: estrutura organizacional, decisões técnicas e sequência de entrega.

Os protótipos navegáveis (chat + orquestração, multi-UF, widget embedded, marketplace, agente real) já provaram a tese visualmente. Esse plano responde à próxima pergunta: quem constrói isso de verdade, com quê, e em que ordem.

Princípio que guia este plano: reaproveitar o que o Bradesco já tem — Apigee no benchmark de gateway, CIAM em Azure AD, squads de CaaS/LaaS em andamento, expertise em Java/Spring Boot — em vez de propor uma stack paralela. Cada decisão abaixo testa primeiro "dá pra fazer com o que já existe?" antes de sugerir algo novo.

As novas variáveis: telemetria, modelo preditivo e Fleet Engine

SageMaker ≠ Bedrock
ML preditivo é uma disciplina separada da IA generativa
Risco de falha mecânica e score de prêmio de seguro são problemas de ML clássico (séries temporais, regressão/classificação sobre dados de telemetria) — não geração de texto. Por isso entram via AWS SageMaker, não Bedrock. O agente generativo continua sendo a camada conversacional: ele consulta o score do SageMaker como mais uma ferramenta, não recalcula nada.
Google Fleet Engine
Fonte de dados, não decisão
Fleet Engine rastreia veículo e otimiza rota — ele fornece a matéria-prima (localização, km rodado, padrão de uso). A previsão de manutenção e o cálculo de seguro são construídos pelo Bradesco por cima desses dados. Recomendo integrar via proxy no Apigee, não direto do domain service — mantém o mesmo ponto único de governança das demais integrações externas.
Natureza diferente de fluxo: telemetria é ingestão contínua (streaming), não request/response pontual como o resto da arquitetura. Vale considerar AWS Kinesis ou IoT Core como camada de entrada antes do vaas-telemetria-service — isso é matéria de ADR específico, não decisão implícita.

Por que Apigee + AWS Bedrock, especificamente

Apigee
Já suporta MCP nativamente
Desde dez/2025, qualquer API já publicada no Apigee pode virar um MCP proxy sem reescrever nada — basta declarar o basepath /mcp e o spec OpenAPI. O proxy é registrado automaticamente no API Hub e herda as mesmas políticas de auth/quota/segurança que toda API já tem hoje. Isso é literalmente reaproveitar a base de APIs que vocês já têm publicada, não construir um gateway de IA do zero.
AWS Bedrock
Modelo + RAG dentro do perímetro AWS
Bedrock hospeda modelos (incluindo Claude) dentro da conta AWS do Bradesco — com VPC, IAM e CloudTrail nativos, em vez de chamar a API do provedor direto pela internet. Bedrock Knowledge Bases é RAG totalmente gerenciado (ingestão, embeddings, retrieval, citação de fonte) — substitui a base manual "regra_tributaria_UF" que simulamos nos protótipos por um serviço gerenciado. AgentCore Gateway expõe os MCP servers ao agente de forma governada.
Java/Spring Boot entra onde sempre entrou: os domain services (consulta de veículo, cálculo de tributo, execução de pagamento) continuam Spring Boot — exatamente o padrão que vocês já operam. O que muda é que cada serviço, ao ser publicado no Apigee, ganha automaticamente uma versão "MCP tool" sem código extra. Vale avaliar Spring AI (projeto da comunidade Spring com suporte a Bedrock, tool-calling e RAG) como camada de integração no lado Java — mas confirme maturidade e versão atual antes de adotar como padrão, é um projeto que evolui rápido.

ADRs a Abrir

Decisões que precisam virar registro formal antes (ou durante) o spike técnico. Template: Contexto → Decisão → Alternativas consideradas → Consequências → Status.

ADR-001Escolha de AI/MCP Gatewayaberto
Apigee (MCP nativo, reaproveita investimento) vs. gateway dedicado (Kong, TrueFoundry) avaliado no benchmark anterior. Critério decisivo: cobertura de governança tool-level vs. esforço de adoção.
ADR-002Hospedagem de modeloaberto
AWS Bedrock (dentro do perímetro AWS) vs. chamada direta a provedor (Anthropic/OpenAI/Google). Critério decisivo: requisito de soberania de dado e auditoria via CloudTrail.
ADR-003Estratégia de RAGaberto
Bedrock Knowledge Base gerenciado vs. pipeline próprio (vector DB self-hosted). Critério decisivo: quem cura e atualiza a base de regras tributárias por UF — isso é o investimento real, não a tecnologia.
ADR-004Modelo de orquestração do agenteaberto
Modelo B do showcase original — agente decide a sequência via MCP, mas execução financeira sempre passa por gate determinístico local, nunca decisão do modelo.
ADR-005Framework de integração Javaaberto
Spring AI vs. SDK AWS direto (bedrock-runtime). Avaliar maturidade da versão atual antes de comprometer como padrão de squad.
ADR-006Estratégia de cloudaberto
Híbrida — core bancário permanece na infraestrutura atual; camada de IA roda em AWS. Reconfirmar contra a recomendação anterior de Azure (CIAM já adotado) — pode haver convivência das duas, não é exclusivo.
ADR-007Modelo de squadsaberto
1 squad satélite na Fase 1, desdobro em 2 só após validar volume no piloto controlado.
ADR-008Plataforma de modelo preditivoaberto
AWS SageMaker vs. alternativa (ex: modelo treinado fora e servido via endpoint próprio). Critério decisivo: quem mantém o ciclo de retreino e quem é dono da qualidade do modelo de risco.
ADR-009Estratégia de ingestão de telemetriaaberto
Streaming (Kinesis/IoT Core) vs. polling periódico do Fleet Engine. Critério decisivo: frequência real necessária pelo modelo preditivo — nem toda decisão precisa de tempo real.
ADR-010Integração com Google Fleet Engineaberto
Proxy via Apigee (consistente com o resto da arquitetura) vs. integração direta do domain service. Critério decisivo: manter ponto único de auditoria/observability para todas as integrações externas, não só as legadas.

Painel de Frota

O contrato de UI não muda. Os protótipos já são, na prática, a especificação visual e de interação da versão final — o trabalho de produção é trocar o que está por trás de cada chamada, não redesenhar a experiência.
08

Fluxo de Requisição: MFE → Core

O ciclo completo de uma mensagem do cliente até a execução real, hop a hop — com troca de token e guardrail marcados em cada etapa.

Cenário de referência: cliente envia uma mensagem no chat embeddado (MFE) perguntando sobre o pagamento do veículo. Cada passo abaixo mostra quem fala com quem, qual token circula e qual controle é aplicado — para deixar explícito que o token de login do usuário nunca atravessa toda a cadeia, e que toda ação financeira passa por um controle determinístico antes de executar.

1
MFE captura a mensagem
O widget de chat (React, Module Federation) recebe o input do cliente. O JWT de sessão do usuário, já emitido pelo CIAM (Azure AD) no login do app, está disponível no contexto do host.
🔑 JWT de sessão (usuário)
2
BFF recebe e faz Token Exchange
O MFE chama o BFF (Spring Boot, camada fina dedicada à experiência de chat) via HTTPS. O BFF valida o JWT recebido e faz OAuth2 Token Exchange (RFC 8693): troca o token do usuário por um token de serviço com escopo restrito. O token de login nunca segue além deste ponto.
🔑 Token exchange → token de serviço🛡️ Validação de assinatura/expiração
3
BFF → API Gateway (Apigee)
O BFF chama o Apigee com o token de serviço. Apigee aplica autenticação, autorização por API Product/escopo, rate limiting/quota e observability — as mesmas políticas que toda API tradicional já tem hoje.
🛡️ AuthN/AuthZ + quota + logging
4
Apigee roteia: REST tradicional ou MCP
Se for uma chamada direta de API, segue para o domain service normalmente. Se for uma chamada de ferramenta vinda do agente, é roteada para o MCP proxy correspondente — mesma infraestrutura, contrato diferente.
⚙️ Política de roteamento por tipo de chamada
5
AgentCore Gateway monta o contexto e chama o modelo
O AgentCore Gateway (AWS Bedrock) reúne system prompt + lista de ferramentas disponíveis (descobertas via os MCP proxies do Apigee) e envia ao modelo. Antes disso, o Guardrail de entrada roda sobre o prompt do usuário.
🛡️ Guardrail de entrada: filtro de conteúdo, detecção de prompt injection, redação de PII
6
Modelo decide tool_use
O modelo decide qual ferramenta chamar e com quais parâmetros. Essa decisão não executa nada por si só — ela é interceptada pelo AgentCore Gateway, que é quem de fato dispara a chamada real.
⚙️ Decisão do modelo ≠ execução
7
AgentCore Gateway → MCP Proxy (Apigee) → Domain Service
A chamada de ferramenta vira uma requisição HTTP real contra o MCP proxy, que invoca o domain service Java/Spring Boot (ex: vaas-tributos-service) usando autenticação serviço-a-serviço.
🔑 mTLS / OAuth2 client credentials (serviço-a-serviço)
8
Domain Service consulta o RAG, se necessário
Para regras que variam por UF, o domain service consulta o Bedrock Knowledge Base: embedding da query → busca vetorial → chunks relevantes + citação de fonte. Isso entra como contexto adicional — nunca como decisão final.
📚 Retrieval com citação de fonte
9
Gate de Política (determinístico)
Antes de qualquer ação financeira, o domain service aplica uma regra fixa e não-IA: aprovação automática dentro do limite, ou bloqueio com necessidade de validação manual. Esta etapa nunca é decidida pelo modelo.
🛡️ Regra determinística, fora do alcance do modelo
10
Domain Service → Core bancário / Legado
Se aprovado, o domain service chama o Core (conta/pagamento) ou o conector de integração com DETRAN/SEFAZ. Essa chamada é sempre determinística — o modelo nunca tem acesso direto ao Core.
🔑 Token de serviço já restrito ao domain service
11
Resposta sobe a cadeia
Domain Service → MCP Proxy → AgentCore Gateway → Modelo. O modelo recebe o resultado da ferramenta e decide se chama mais alguma ferramenta ou já responde em texto. O Guardrail de saída verifica grounding contra o que as ferramentas retornaram, reduzindo o risco de alucinação.
🛡️ Guardrail de saída: verificação de grounding + sanitização
12
BFF → MFE
O BFF formata a resposta final no contrato de UI do widget e devolve ao MFE, que renderiza a mensagem para o cliente. Nenhum token de acesso ao Core circula de volta — só o conteúdo da resposta.
🔑 Sem token de backend no caminho de volta
Princípio de segurança que atravessa todo o fluxo: o token do usuário morre no BFF (passo 2); a partir daí, tudo é token de serviço com escopo restrito. O modelo decide o quê chamar, nunca chama diretamente — sempre passa pelo AgentCore Gateway e pelo MCP proxy do Apigee. E nenhuma ação financeira sai do gate determinístico do passo 9, por mais convincente que a conversa esteja.
