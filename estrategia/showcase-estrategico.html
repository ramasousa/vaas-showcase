01

Visão Geral

O problema, o tamanho da oportunidade e a tese de produto — antes de entrar em arquitetura.

○ O problema

Hoje, o momento da compra do veículo é o ponto de maior atrito da jornada: o cliente sai do ecossistema bancário para resolver IPVA, emplacamento, transferência de propriedade e seguro em sistemas fragmentados — DETRAN estadual, SEFAZ, despachante, seguradora. O Bradesco financia o bem, mas não captura o resto da jornada — nem o relacionamento, nem a receita, nem o dado.

○ O tamanho da oportunidade

🚗
~15,7 milhões
de veículos usados/seminovos negociados em 2025 no Brasil — recorde histórico (Fenauto), 17%+ de crescimento sobre 2024.
💳
R$ 192,1 bi
em crédito para compra de veículos em 2024 (+36,4%) — 44% dos emplacamentos já são financiados, em tendência de alta.
📈
2,8 milhões
de veículos 0KM projetados para 2025 (Anfavea) — cada um gera IPVA + emplacamento + transferência recorrentes.

É um mercado gigantesco, já maturado para crédito embedded — exatamente a tese que já estamos validando com CaaS e LaaS. O que falta é estender essa tese para o momento de pagamento do bem em si, não só para o crédito que o viabiliza.

○ A tese de produto: uma nova camada — VaaS

Proponho tratar isso não como "mais uma API", mas como uma quinta camada de domínio, paralela a BaaS / CaaS / LaaS: VaaS — Vehicle as a Service. Ela concentra os produtos e integrações específicos do universo veicular (cálculo e pagamento de tributos, emplacamento, transferência, seguro), consome Core Platform e CaaS/LaaS para crédito, e se expõe a parceiros (concessionárias, marketplaces, fintechs de mobilidade) através do BaaS — fechando o ciclo na mesma infraestrutura de APIs que já estamos construindo.

Por que isso é estratégico agora: não estamos criando uma plataforma nova do zero — estamos estendendo Core Platform, CaaS e LaaS com um domínio vertical de alto volume e alta frequência de tributo/taxa, usando a primeira aplicação real de orquestração agêntica dentro do Open Platform.

○ Os três conceitos que sustentam a camada de IA

Você pediu pra organizar isso de um jeito que dê pra explicar sem pressupor conhecimento prévio — então vai direto ao ponto de cada peça:

AI Gateway
O "Axway da IA"
Fica entre as aplicações e os modelos/ferramentas de IA. Centraliza autenticação, custo, rate limit, roteamento entre provedores e auditoria — exatamente o papel que o Axway hoje cumpre para APIs REST. Gartner projeta que 75% dos fornecedores de API Gateway terão recursos de IA nativos até o fim de 2026. Kong e Azure APIM — já no nosso benchmark de gateway — estão entre eles.
MCP
Tool-calling padronizado
Model Context Protocol é o padrão (Anthropic, adotado por OpenAI, Google, Microsoft, AWS) que permite ao agente descobrir e chamar ferramentas — "consultar débito de IPVA", "gerar guia", "agendar vistoria" — sem integração ponto-a-ponto pra cada sistema. Já passa de 9.400 servidores no registro público e 78% de adoção em produção entre times de IA enterprise.
RAG
Buscar a regra, não "adivinhar"
Retrieval-Augmented Generation: antes de responder, o agente busca a regra certa numa base de conhecimento, em vez de gerar a partir de memória do modelo. Essencial aqui porque IPVA, alíquota e prazo de licenciamento variam nas 27 UFs — é a mesma lógica que já prototipamos no Débito Veicular do developer portal, só que agora alimentando uma execução, não uma consulta.
Conexão direta com o que já existe: isso não é uma stack nova — é a extensão natural da proposta de API Platform (Axway → Apigee/Azure APIM) e do trabalho de multi-UF do Débito Veicular. O AI Gateway e o MCP Gateway, na prática, viram mais um produto exposto pelo mesmo gateway de API que já estamos avaliando.
03

Modelos Viáveis de Orquestração

Três níveis de autonomia possíveis. Quanto mais autonomia, mais flexível — e mais peso de governança precisa carregar.

Modelo A — Workflow Determinístico + IA Assistiva
Baixa autonomia · aprovação de compliance mais rápida
Conservador

A regra de negócio vive num motor de orquestração clássico (BPM / state machine). A IA aparece só na conversa com o cliente — explica o cálculo, tira dúvida, nunca decide a sequência de chamadas.

Vantagens
  • Auditável por padrão, sem "caixa-preta" de decisão
  • Mais fácil de aprovar com Compliance/Risco
  • Latência previsível
Limites
  • Cada novo estado/parceiro = novo fluxo codificado
  • Não escala bem para 27 variações de UF
  • Pouco diferencial competitivo
Modelo B — Agente Orquestrador com MCP Governado
Autonomia média · gate determinístico na execução financeira
Recomendado p/ começar

O agente decide qual sequência de ferramentas chamar (via MCP Gateway) e a base RAG fornece a regra de cada UF. Mas a execução financeira sempre passa por um gate determinístico — limite de valor, dupla validação, política fixa — que não é decidido pelo modelo.

Vantagens
  • Flexível para novos estados sem reescrever fluxo
  • Risco de execução contido pelo gate
  • Trilha de auditoria nativa do MCP Gateway
Limites
  • Exige curadoria contínua da base RAG por UF
  • Operação nova para os times — curva de aprendizado
Modelo C — Mesh Multiagente Especializado
Alta autonomia · visão de 12–18 meses
Horizonte

Agentes especializados por domínio — tributário, licenciamento, crédito, pagamento — coordenados por um agente central. É o padrão que bancos já usam hoje em AML/compliance (Deloitte, IMF), aplicado aqui ao universo veicular.

Vantagens
  • Mais extensível — novo domínio = novo agente, não reescrita
  • Paralelismo real entre tarefas
Limites
  • Maior complexidade operacional e de observabilidade
  • Só faz sentido depois de provar o Modelo B em produção
Recomendação: começar no Modelo B com 1–2 estados-piloto, provar o gate de governança e a curadoria RAG, e só então evoluir para o Modelo C onde o volume de domínios justificar.

Latência & Risco Técnico

Estimativas de ordem de grandeza para calibrar expectativa — precisam ser validadas com spike técnico, não são SLA.

○ Budget de latência por etapa

EtapaEstimativaObservação
Chamada ao AI Gateway / modelo300–800 msDepende do modelo e se a resposta é em streaming
Retrieval na base RAG50–150 msBaixo, mas cresce com base mal indexada
Chamada MCP a sistema legado (DETRAN/SEFAZ)500 ms – vários seg.Maior fonte de risco — 27 UFs com maturidade desigual de API
Jornada completa (calcular + gerar guia)~2–5 s cenário bomPode passar de 10s se o Estado estiver lento — desenhar como assíncrono, nunca bloquear o checkout do veículo

○ Catálogo de riscos técnicos

Alto
Heterogeneidade dos DETRANs
27 integrações estaduais, algumas sem API moderna. Mitigação: avaliar agregador/despachante digital para cobertura inicial em vez de integração direta estado a estado.
Alto
Governança de execução financeira
Pagar tributo em nome do cliente não pode ser "decisão livre" do agente. Mitigação: gate determinístico + trilha de auditoria imutável no MCP Gateway (padrão já exigido por bancos hoje).
Médio
Alucinação de regra tributária
Modelo "inventa" alíquota ou prazo. Mitigação: RAG bem curado + validação contra fonte oficial — nunca cálculo de memória do modelo.
Médio
Exposição de credenciais legadas
Agente nunca deve enxergar credencial de sistema legado. Mitigação: MCP Gateway isola credenciais via OAuth2/cofre — é literalmente a função central do gateway.
Médio
Drift regulatório por UF
Regra estadual muda e a base RAG fica desatualizada. Mitigação: tratar atualização da base como item de catálogo governado — mesmo padrão do BaaS Architecture Playbook.
Médio
Concentração em parceiro externo
Se a cobertura DETRAN depender de um único despachante/agregador, isso cria risco de concentração e de negociação. Mitigação: contrato com cláusula de portabilidade + roadmap de internalizar os estados de maior volume.

Modelo de Execução

Como provar o conceito sem comprometer um squad inteiro antes de validar a tese.

○ Fase 0 — Squad satélite (6–8 semanas)

Em vez de criar um 5º squad formal de imediato, propor um squad satélite cross-funcional, com gente "emprestada" das estruturas já existentes, para validar o Modelo B com 1–2 estados antes de comprometer recurso permanente.

Times internos envolvidos
Open APIs — exposição de API e MCP servers (seu time)
Core Platform — BFF, autenticação, pagamentos
Arquitetura (time de Arquitetura) — padrão de AI Gateway corporativo
CaaS / LaaS — encaixe do crédito embedded no fluxo
Parceiros externos a avaliar
Despachante digital / agregador — cobertura DETRAN sem integrar 27 estados direto
Fornecedor de AI/MCP Gateway — Kong, Azure (CIAM já adotado) ou alternativa enterprise do benchmark
Consultoria de multiagente — só se/quando avançar para o Modelo C

○ Governança

Não criar ritual novo — encaixar nos que já existem: API Review Board passa a avaliar novos MCP tools como avalia hoje novas APIs; Weekly Cross-Squad Sync ganha um item fixo de "VaaS / Mobilidade" enquanto for spike.

○ Cloud híbrida ou um único vendor?

Dado o contexto regulatório (Bacen, LGPD, soberania de dado de crédito) e que já existe investimento em Azure AD/CIAM, a recomendação é híbrida — não fechar num único vendor agora:

Permanece
Core bancário
Dados sensíveis de crédito e pagamento ficam na infraestrutura privada atual do Bradesco — não migram para a camada de IA.
Camada de IA
Gateway, agente, RAG
Pode rodar em cloud pública com fronteira de dados estrita, aproveitando o CIAM em Azure já adotado e, possivelmente, o Azure APIM com capacidades de AI/MCP Gateway já no nosso benchmark.
Anti-lock-in
Deployment in-VPC
Priorizar gateways com suporte a deployment in-VPC / air-gapped — vários fornecedores enterprise de MCP Gateway já oferecem isso justamente para banco regulado.
Coerência com o que já documentamos: isso segue o mesmo princípio de "abertura de stack" do BaaS Architecture Playbook — não fixamos tecnologia nova como requisito, avaliamos contra o que já está no radar do benchmark de gateway (Azure APIM como favorito contextual, Kong como alternativa).
